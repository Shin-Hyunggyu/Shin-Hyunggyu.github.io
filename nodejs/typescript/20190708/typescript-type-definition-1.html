type definition 1

1. type 을 정의해야 하는 이유?

npm 에서 받을 수 있는 대부분의 3rd Party 모듈들은 js 로 작성되있다. 이 모듈들을 TypeScript 프로젝트에 import 해 오면 TypeScript Compiler 가 에러를 내뿜기 시작한다. 원인은 이 모듈에 대한 타입 정보가 없어 어떻게 구성되어 있는지, 이 모듈이 무엇인지를 알 수 없기 때문이다. 그래서 TypeScript Compiler 에게 이 모듈들에 대한 타입 정보를 알려 줄 필요가 생겼다. 그 외에도 type 과 interface 같은 타입 정의를 외부에 두어 조금 더 구조적이고 클린한 코드를 생산할 수 있다.

TypeScript Compiler 는 *.d.ts 파일을 통해 타입 정보를 탐색할 수 있으며 기본적으로 node_modules/@types 를 탐색하도록 되있다. 이 경로는 tsconfig.js 를 통해 변경이 가능하다.

2. @types

대부분의 유명한 모듈들은 @types 를 통해 타입 정보를 배포하고 있다. 그러지 못한 모듈들은 개발자가 직접 타입 정의를 할 수 밖에 없다.

npm install --save @types/모듈명

3. reference

TypeScript Compiler 가 탐색할 수 있는 범위에 *.d.ts 파일이 존재한다면 모든 *.ts 파일에 타입이 적용될 수 있지만 외부에 *.d.ts 파일이 존재한다면 아래와 같은 참조 선언을 통해 직접 *.ts 파일에 타입을 적용 시킬 수 있다. 참조 선언은 파일 최상위에 위치해야 한다.

/// <reference path=“location/definition.d.ts” />

4. declare

declare 는 ambient declaration 시 사용하는 키워드이다. ambient 는 ‘주변’ 이라는 뜻으로 TypeScript Compiler 에게 ‘프로젝트 어디엔가 이런게 존재하고 있어!’ 라고 해석될 수 있다. declare 는 주로 *.ts 나 *.js 파일에서 생성되지 않은 global 변수를 선언할 때 사용하는데, 예를 들면 브라우저에서 사용하는 document, window 객체나 jQuery 에서 사용하는 $ 객체, NodeJS 에서 사용하는 os, process 객체같은 것들이 있다. 변수뿐만 아니라 module, namespace, interface, class, function 등등 다양한 타입의 선언이 가능하다.

declare const $: JQuery;

declare function require(moduleName: string): any;

// Top-level 에서만 사용이 가능 (두번째 declare 사용불가)
declare module 'person' {
  declare const name: string;
  const age: number;
}

- declare 와 export

Top-level 에서 declare 와 export 는 비슷한 해석을 가진다. declare module 을 제외한 모든 declare 선언을 export 키워드로 바꿔도 동일한 정의가 가능하다. 하지만 declare 는 export 보다 좀 더 넓은 범위의 타입 정의라고 할 수 있다. export 가 단순히 내보내기를 위한 키워드라면 declare 는 TypeScript Compiler 에게 알려주는 키워드이다.

Top-level 에서 declare module 선언과 export 키워드를 함께 사용했더니 import 할 때 모듈을 찾지 못하는 ‘cannot find module’ 에러가 발생하였다. Top-level 에서 declare 선언과 export 키워드를 동시에 사용하지 말자.

- type 과 interface

Top-level 에서 정의한 type 과 interface 는 declare 나 export 를 앞에 붙이지 않아도 TypeScript Compiler 가 잘 인식한다. assist 또한 가능하다. type 은 하나 또는 여러 타입들의 alias 라고 하면 interface 는 새로운 타입을 만드는 것이다. 에러 출력 시에는, type 은 alias 된 타입들을 세부적으로 보여 주지만 interface 는 interface 명만 보여준다.

declare interface IPerson {
  name: string;
  age: number;
}

// declare 선언과 동일
interface IPerson {
  name: string;
  age: number
}

- module 과 namespace

module 과 namespace 는 독립적인 공간 안에 선언과 코드를 작성한다는 점에서 비슷하다. 하지만 사용하는 용도나 정의하는 방법이 서로 다르며, 가장 큰 차이점은 module 정의는 모듈 로더에 대한 의존성을 가지고 있다는 것이다.

module 정의는 import 나 require 같은 키워드를 통해 불러 올 수 있는 객체를 정의하는 일이다. 모듈명은 문자열로 작성해야 하며 경로 형식도 가능하다. TypeScript Compiler 는 처음에 적절한 경로로 모듈을 탐색하고 모듈을 찾지 못하면 ambient declaration 에서 찾는다.

// 모듈 선언
declare module 'person' {
  export const name: string;
  export const age: number;
}

// 짧은 모듈 선언 (자식들은 모두 any 타입을 가짐)
declare module 'person';

// 경로 형식의 모듈 선언
declare module '@material-ui/core' {
  export const CustomPropTypes: 'none' | 'dense' | 'normal';
}

// 모듈 사용
import person from 'person';
import { Button } from '@material-ui/core';

const name = person.name;
const age = person.age;

namespace 정의는 global 네임스페이스에 정의된 객체라고도 불린다. 그래서 <script> 태그를 통해 선언된 global 변수(객체)를 구조화하여 정의하는데 좋은 방법이 될 수 있다.

// 네임스페이스로 jQuery 정의
declare namespace $ {
  export function ajax(url: string, settings?: Settings);
  export function map(array: Array, callback: function);
}

// 네임스페이스 사용
$.ajax('https://lofty87.github.io/');
$.map([ 'name', 'age' ], val => `public ${val}`);

기존에 declare module 로 정의된 모듈을 export namespace 로도 바꿀 수 있다. 하지만 모듈을 불러오기 위해서는 모듈명이 아닌 경로를 통해 불러와야 하며, 코드 작성 시 감싸져 있는 네임스페이스로 하여금 개발자를 불편하게 만든다.

// 모듈로 정의했던 jQuery 를 네임스페이스로 정의
export namespace $ {
  export function ajax(url: string, settings?: Settings);
  export function map(array: Array, callback: function);
}

// 불필요하게 $ 로 감싸고 있는 판국
import $ from '../@types/jQuery';

$.$.ajax('https://lofty87.github.io/');
$.$.map([ 'name', 'age' ], val => `public ${val}`);

네임스페이스 명은 변수로 취급되기 때문에 문자열이 아닌 일반 식별자로 작성해야 한다. 또 TypeScript Compiler 는 네임스페이스 안에 작성된 코드가 없으면 그 네임스페이스를 무시하므로 내부에 꼭 코드를 작성하도록 하자.

// 인식 불가
declare namespace $ {}

5. 내부에 모듈 타입 정의

TypeScript Compiler 는 실제 모듈을 불러오는 방법과 똑같이 인식하기 때문에 아래와 같이 definition 파일을 둘 경우 해당 모듈의 타입을 정의할 수 있다. 구지 declare module 을 통해 직접 모듈의 위치를 알려 줄 필요도 없다.

  • 모듈이 파일명일 때
    • src/person.js
    • src/person.d.ts
  • 모듈이 폴더명일 때
    • src/person/index.js
    • src/person/index.d.ts
// 내부에 모듈 타입 정의
interface IPerson {
  name: string;
  age: number;
}

export const person: IPerson;

export default person;

// 모듈 사용
import person from './person';

const name = person.name;
const age = person.age;

6. 외부에 모듈 타입 정의

외부에 모듈 타입을 정의하는 일은 보통 3rd Party js 모듈이 타입 정의를 제공하지 않거나 잘못된 타입 정의를 수정하기 위해 사용한다. 외부에 모듈 타입을 정의하는 방법은 내부에 모듈 타입을 정의하는 방법과 비슷하다. 단지, 내부가 아닌 외부에 모듈 타입 정의가 있을 뿐이다. 폴더명 또는 파일명을 모듈명과 동일하게 만들고 TypeScript Compiler 가 탐색할 수 있도록 tsconfig.js 에 typeRoots 값을 추가하면 된다.

  • src/@types/person.d.ts
  • src/@types/person/index.d.ts

주의할 것은 이미 타입 정의가 제공되고 있는데 중복으로 정의할 경우 TypeScript Compiler 가 두개로 구분해 인식하여 원하는 타입 정의가 나오지 않을 수 있다. 따라서 최대한 중복되지 않도록 신경써야 한다. 또 폴더명으로 모듈을 정의할 때 index.d.ts 가 없으면 TypeScript Compiler 가 에러를 출력하므로 코드를 작성하지 않더라도 반드시 추가하도록 한다.

또 다른 방법으로 정의하고 싶다면 declare module 을 사용해도 된다. TypeScript Compiler 가 탐색 범위에 모듈 정의가 있다면 자동으로 인식할 것이며, 범위가 아니더라도 reference 를 통해 개발자가 직접 인식시키면 된다.

// 모듈 정의
declare module 'person' {
  export const name: string;
  export const age: number;
}

// 모듈 정의를 직접 참조시키기
/// <reference path=“../others/person/index.d.ts” />
import person from 'person';

const name = person.name;
const age = person.age;

- 외부에 타입을 정의하려면?

Typescript Compiler 가 탐색할 수 있도록 tsconfig.js 에 typeRoots 값을 추가해 주어야 한다. 값을 설정하지 않으면 기본적으로 node_modules/@types 만 탐색하도록 되있다.

{
  "typeRoots": [ "node_modules/@types", "src/@types" ]
}

- 외부에 타입을 정의하여 기존 타입 정의를 오버라이딩 하려면?

이리저리 구글링하고 테스트 해보았지만 한마디로 불가능하다. Typescript Compiler 는 동일한 부분을 오버라이딩 하는 것이 아니라 두개로 인식할 뿐이다. 정말 불가피한 상황이라면 직접 타입 정의를 수정하거나 오버라이딩이 필요한 타입 정의들만 복사해 수정하고 외부에 타입을 정의하면 된다. 당연히 기존 타입 정의들은 중복되지 않도록 지우는 것이 좋다. 단, 예외적으로 오버라이딩 효과를 낼 수 있는 경우가 있는데 그것은 완전히 서로 다른 내용을 가진 같은 명의 모듈 정의일 경우이다. 오버라이딩이라기 보다는 병합에 가까운 수준이다.

// @material-ui/core 모듈에는 PropTypes 라는 타입이 이미 정의되어 있다.
import { PropTypes } from '@material-ui/core';

// 중복 정의를 통해 오버라이딩 하지 말아야 할 경우 (PropTypes 겹침)
declare module '@material-ui/core' {
  export const PropTypes: 'none' | 'dense' | 'normal';
}

// 중복 정의를 통해 오버라이딩 효과를 낼 수 있는 경우
declare module '@material-ui/core' {
  export const CustomPropTypes: 'none' | 'dense' | 'normal';
}

- 타입을 정의할 때 유의할 점

*.ts 코드와 *.d.ts 타입 정의가 정확하게 매칭되어야 한다. 정확하게 매칭되지 않을 경우 혹시나 TypeScript Compiler 린팅은 통과했지만 런타임 시 에러가 발생할 가능성이 있다. TypeScript Compiler 는 타입 정의를 통해서만 코드를 해석하므로 개발자가 고의로 타입을 속일 수도 있다. 또 타입 정의에 집중한 나머지 type 이나 interface 를 export 하는 실수도 할 수 있으니 조심하도록 하자.

// person.js
const person = {
  name: 'lofty87',
  age: 33
};

export default person;

// person.d.ts
interface IPerson {
  name: string;
  age: number;
}

export const person: IPerson;

export default person;

// person.d.ts (export 실수한 경우)
interface IPerson {
  name: string;
  age: number;
}

export default IPerson;
