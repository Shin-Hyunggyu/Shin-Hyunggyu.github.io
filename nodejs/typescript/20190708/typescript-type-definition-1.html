type definition 1

1. type 을 정의해야 하는 이유?

npm 에서 받을 수 있는 대부분의 3rd Party 모듈들은 js 로 작성되어 있다. TypeScript 프로젝트에서 import 를 통해 이 모듈들을 불러오면 TypeScript Compiler 가 에러를 내뿜기 시작한다. 원인은 이 모듈에 대한 타입 정보가 없어 어떻게 구성되어 있는지, 이 모듈이 무엇인지를 알 수 없기 때문이다. 그래서 TypeScript Compiler 에게 이 모듈들에 대한 타입 정보를 알려 줄 필요가 생겼다. 그 외에도 type, interface 같은 타입 정의를 통해 조금 더 효율적이고 클린한 코드를 생산할 수 있다.

TypeScript Compiler 는 *.d.ts 파일을 통해 타입 정보를 탐색할 수 있다. TypeScript 는 기본적으로 node_modules/@types 를 탐색해 모듈들의 타입 정보를 얻는다. (tsconfig.js 를 통해 변경가능)

2. @types

대부분의 유명한 모듈들은 @types 를 통해 타입 정보를 배포하고 있다. 하지만 그러지 못한 모듈들은 개발자가 직접 타입 정의 파일을 작성해 사용할 수 밖에 없다.

npm install --save @types/모듈명

3. declare keyword

declare 는 TypeScript Compiler 에게 ‘프로젝트 어디엔가 이런게 존재하고 있어!’ 라고 알려주는 역할을 한다. declare 는 module 뿐만 아니라 const, let, class, function 앞에서도 사용이 가능하며 오직 Top-level 에서만 사용할 수 있다.

// 'person' 이라는 모듈이 존재하고 있어!
declare module 'person' {}

// declare 사용가능
declare const name: string;

// declare(두번째) 사용불가
declare namespace 'person' {
  declare const name: string;
}

- declare 와 export

Top-level 에서 타입을 정의하는데 있어 declare 와 export 는 비슷한 해석을 가진다. declare 대신 export 를 사용해도 TypeScript Compiler 가 인식할 수 있고 assist 도 가능하다. 하지만 사용하는 의미와 용도가 달라 외부에 타입을 정의할 때에는 declare 를 그 외에는 export 를 사용하는 것이 좋겠다.

- type 과 interface

Top-level 에서 정의한 type 과 interface 는 declare 나 export 를 앞에 붙이지 않아도 TypeScript Compiler 가 인식할 수 있고 assist 도 가능하다.

- module 과 namespace

타입을 정의하는데 있어 module 과 namespace 는 독립적인 공간을 말한다. module 대신 namespace 를 사용해도 100% 같은 효과를 가진다. 하지만 사용하는 의미와 용도가 달라 모듈을 정의할 때는 module 을 독립적인 공간을 정의할 때는 namespace 를 사용하는 것이 좋겠다.

4. 내부 타입 정의

TypeScript Compiler 는 실제 모듈을 불러오는 방법과 똑같이 경로를 인식하기 때문에 아래와 같이 definition 파일을 둘 경우 해당 모듈의 타입을 정의할 수 있다.

  • 모듈이 파일명일 때
    • src/person.js
    • src/person.d.ts
  • 모듈이 폴더명일 때
    • src/person/index.js
    • src/person/index.d.ts
interface IPerson {
  name: string;
  age: number;
}

export const person: IPerson;

export default person;

내부 타입 정의는 실제 모듈을 불러오는 경로와 똑같기 때문에 구지 declare 를 통해 모듈의 위치를 TypeScript Compiler 에게 알려 줄 필요가 없다. person 변수 앞에 export 를 붙여 준 이유는 TypeScript Compiler 가 Top-level 안에 평범한 변수 정의를 제한하기 때문이다. type 과 interface 를 제외한 평범한 변수를 Top-level 안에 정의할 때에는 앞에 export 나 declare 를 붙여 주도록 하자. block 안, 예를 들면 module 이나 namespace 안에서는 정의가 가능하다.

5. 외부 타입 정의

외부 타입 정의는 보통 3rd Party js 모듈이 타입 정의를 제공하지 않거나 잘못된 타입 정의를 수정하기 위해 사용한다. 아래와 같이 파일명이나 폴더명을 모듈명으로 한다고 해서 TypeScript Compiler 가 해당 명의 모듈 타입 정보로 인식하지 않는다. 반드시 declare 를 통해 모듈을 정의해야 한다.

  • src/@types/person.d.ts
  • src/@types/person/index.d.ts

주의할 것은 이미 타입 정의가 제공되고 있는데 중복으로 정의할 경우 TypeScript Compiler 가 두개로 구분해 인식하여 원하는 타입 정의가 나오지 않을 수 있다. 따라서 최대한 중복되지 않도록 해야 정신건강에 이롭다.

// 일반적인 모듈명
declare module 'person' {
  export const name: string;
  export const age: number;
  export type info = {
    address: string
  };
}

// 경로 형식의 모듈명도 가능
declare module '@material-ui/core' {
  export const CustomPropTypes: 'none' | 'dense' | 'normal';
}

- 외부 타입을 정의하려면?

Typescript Compiler 가 탐색할 수 있도록 tsconfig.js 에 typeRoots 값을 추가해 주어야 한다. 값을 설정하지 않으면 기본적으로 node_modules/@types 만 탐색하도록 되있다.

{
  "typeRoots": [ "node_modules/@types", "src/@types" ]
}

- 외부 타입 정의를 통해 기존 타입 정의를 오버라이딩 하려면?

이리저리 구글링하고 테스트 해보았지만 한마디로 불가능하다. Typescript Compiler 는 동일한 부분을 오버라이딩 하는 것이 아니라 두개로 인식할 뿐이다. 정말 불가피한 상황이라면 직접 타입 정의를 수정하거나 오버라이딩이 필요한 타입 정의들만 복사해 수정하고 외부 타입으로 정의하면 된다. 당연히 기존 타입 정의들은 중복되지 않도록 지우는 것이 좋다. 단, 예외적으로 오버라이딩 효과를 낼 수 있는 경우가 있는데 그것은 완전히 서로 다른 내용을 가진 같은 명의 모듈 정의일 경우이다. 오버라이딩이라기 보다는 병합에 가까운 수준이다.

// @material-ui/core 모듈에는 PropTypes 라는 타입이 이미 정의되어 있다.
import { PropTypes } from '@material-ui/core';

// 중복 정의를 통해 오버라이딩 하지 말아야 할 경우 (PropTypes 겹침)
declare module '@material-ui/core' {
  export const PropTypes: 'none' | 'dense' | 'normal';
}

// 중복 정의를 통해 오버라이딩 효과를 낼 수 있는 경우
declare module '@material-ui/core' {
  export const CustomPropTypes: 'none' | 'dense' | 'normal';
}

- 타입을 정의할 때 유의할 점

*.ts 코드와 *.d.ts 타입 정의가 정확하게 매칭되어야 한다. 정확하게 매칭되지 않을 경우 혹시나 TypeScript Compiler 린팅은 통과했지만 런타임 시 에러가 발생할 가능성이 있다. TypeScript Compiler 는 타입 정의를 통해서만 코드를 해석하므로 개발자가 고의로 타입을 속일 수도 있다. 또 타입 정의에 집중한 나머지 type 이나 interface 를 export 하는 실수도 할 수 있으니 조심하도록 하자.

// person.js
const person = {
  name: 'lofty87',
  age: 33
};

export default person;

// person.d.ts
interface IPerson {
  name: string;
  age: number;
}

export const person: IPerson;

export default person;

// person.d.ts (export 실수한 경우)
interface IPerson {
  name: string;
  age: number;
}

export default IPerson;
